## Stage (Layer) ordering

The Containerfile is a multi-stage build. Each `FROM ... AS <stage>` is a
stage; within a stage, numbered **sub-layers** (`Layer N-M`) group related
`RUN` / `ARG` / `USER` directives. The table below reflects the file as of
2026-06-29.

| Stage (`FROM ... AS`) | Sub-layer | Directive(s) | Purpose | Inputs |
|---|---|---|---|---|
| `manjarolinux/base:latest` | 0-1 | - | Base image | - |
| `base` (`manjarolinux/base:latest`) | 1-1 | `ARG` | Receive build-args from the outer environment. | `HOST_UID`, `HOST_GID`, `USERNAME` |
| `base` | 1-2 | `COPY bind/layer_1_files/pacman_mirrorlist /etc/pacman.d/mirrorlist`, `COPY --from=deps layer_1/pacman.txt /tmp/pacman_deps.txt`, then `RUN --mount=type=cache,target=/var/cache/pacman/pkg pacman -Sy ... $(sed ... \| xargs)` (as root) | Seed a Japan-optimized pacman-mirrors custom mirrorlist from the build context, then install the full Layer 1 pacman set (incl. `zsh`) from `dependencies/layer_1/pacman.txt` (generated by `make gen-deps`; spec 20 I5/I8). pacman's package cache is backed by a BuildKit cache mount so cache-miss rebuilds reuse downloaded tarballs without re-downloading and with no image bloat. | `bind/layer_1_files/pacman_mirrorlist`; `dependencies/layer_1/pacman.txt` via named BuildKit context `deps`; cache mount `/var/cache/pacman/pkg` |
| `base` | 1-3 | `RUN groupmod/usermod ...` | Remap the base image's existing `builder` user/group to the host uid/gid and rename it to `${USERNAME}` when set; set the login shell to `/usr/bin/zsh`. Manjaro base ships `builder` at 1000:1000, so uid/gid only change when they differ from the host. Runs as root; the `USER` switch is deferred to 1-4. | `HOST_UID`, `HOST_GID`, `USERNAME` |
| `base` | 1-4 | `RUN set -e; ...` then `USER ${USERNAME}` | UID-collision fallback + sudoers, then switch to the non-root user. If `${HOST_UID}` already exists, adopt that account (fix shell + home); otherwise create the user/group fresh. Ensures `/home/${USERNAME}` exists with `HOST_UID:HOST_GID` / `0755`. Writes `/etc/sudoers.d/dotfiles` granting `${ACTUAL_USER} ALL=(ALL) NOPASSWD: ALL` at `0440` so `paru` / `makepkg` can run non-interactively. Runs as root (writing sudoers requires root ‚Äî it cannot be bootstrapped via `sudo` before it exists); `USER ${USERNAME}` follows so later stages run non-root. | `HOST_UID`, `HOST_GID`, `USERNAME` |
| `no-config-base` (`FROM base`) | 2 | _(TODO, currently empty)_ | Secret-free staging stage. `chezmoi apply` is **not** run here at build time ‚Äî per [`13-secret-management.md`](13-secret-management.md) ¬ß5 it runs at container start / interactive session. Also the planned home for `pacman` config. (`base-devel` is already installed in Layer 1-2, inherited via `FROM base`.) | (none at build ‚Äî runtime `BW_SESSION` only) |
| _(planned)_ `tooling` | 3+ | ‚Äî | Install per-layer tool sets (`paru`, `nix`, `mise`, `uv`) per `dependencies/layer_<N>/<manager>.txt`. Not yet implemented. | `dependencies/layer_<N>/<manager>.txt` (generated from `dependencies/packages.toml`) |

### Notes on the current state

- The `base` stage is the only fully-implemented stage today. It performs the
  full user-remap / sudoers setup and installs the Layer 1 pacman set from
  `dependencies/layer_1/pacman.txt` (realizing I5 / I8 in
  [`20-container-rules.md`](20-container-rules.md) for the pacman/layer-1
  case). `paru` / `nix` / `mise` / `uv` and the planned `tooling` stage
  remain forward-looking.
- `no-config-base` exists as a named stage (`FROM base AS no-config-base`)
  but its body is a TODO comment only. `chezmoi apply` is **not** wired into
  the build ‚Äî per [`13-secret-management.md`](13-secret-management.md) ¬ß5 it
  runs at runtime (container start), keeping the image secret-free.
- Layer 1-3 and Layer 1-4 together realize invariant **I7** of spec 20
  (`builder` remapped to `${USERNAME}` is the only non-root account; `USER`
  is set before any installation step that does not require root). The
  `USER ${USERNAME}` directive sits AFTER 1-4 (at the end of the `base`
  stage): 1-4's `RUN` executes as root because writing `/etc/sudoers.d/`
  requires root (sudoers cannot be bootstrapped via `sudo` before it exists);
  `USER` is then set so `no-config-base` and the future `tooling`/`paru`
  stages run non-root.

## Acceptance criteria

A new stage may land only when:

1. its name appears in this spec under "Stage (Layer) ordering"
2. its `dependencies/layer_<N>/<manager>.txt` is generated, not hand-written
3. the corresponding [`02-installed-programs.md`](02-installed-programs.md) entries are reachable from `packages.toml`
4. invariants I6‚ÄìI8 in [`20-container-rules.md`](20-container-rules.md) hold after the change

## Open questions

- Q1: how are AUR packages installed without an interactive sudo password inside the build? **Partly resolved:** Layer 1-4 now provisions `/etc/sudoers.d/dotfiles` with `NOPASSWD: ALL` for the remapped user, so `paru` / `makepkg` can run non-interactively. The remaining open parts are (a) whether AUR builds happen in a build stage or at container start, and (b) whether the `paru`/AUR cache is also backed by a `--mount=type=cache` ‚Äî the **pacman** cache already is (Layer 1-2), so only the AUR/`paru` cache is undecided.
- Q2: ~~does `chezmoi apply` live in the `no-config-base` build stage (apply once, bake into image) or in the container entrypoint (apply at every `up`)?~~ **Resolved:** the entrypoint / runtime. `chezmoi apply` runs at container start, never during `make build`; the image stays secret-free (see [`13-secret-management.md`](13-secret-management.md) ¬ß5).

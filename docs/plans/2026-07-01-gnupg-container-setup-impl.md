# `gnupg` in the container (named volume at `GNUPGHOME`) — Implementation Plan

**Status:** pending
**Spec:** [`docs/specifications/implementations/2026-07-01-gnupg-container-setup-design.md`](../specifications/implementations/2026-07-01-gnupg-container-setup-design.md)
**Parent issue:** [`docs/issues/2026-07-01-gnupg-container-setup.md`](../issues/2026-07-01-gnupg-container-setup.md)
**Review trail:** conversational approval 2026-07-01 (lightweight path; no spec-09 letter pass — the change preserves the existing secret-free invariant and introduces no new secret transport; the Bitwarden key-import secret flow is deferred to [`2026-07-01-gnupg-bitwarden-import.md`](../issues/2026-07-01-gnupg-bitwarden-import.md))

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give the container a working, persisted `gnupg` setup that honors the existing `GNUPGHOME=$XDG_DATA_HOME/gnupg` (from `dot_zshenv.tmpl`), by installing `gnupg` + `pinentry` (Layer 1 pacman), baking `~/.local/share/gnupg` (`0700`, owner-correct) as a named-volume mountpoint, mounting the `dotfiles_gnupg` Podman named volume there, and excluding the keyring from chezmoi — without baking any key material into image layers.

**Architecture:** No new build stage. `gnupg` / `pinentry` enter via the existing Layer 1 pacman set (generated from `packages.toml`). Containerfile Layer 1-5 gains a sibling `install -d -m 0700` for the gnupg mountpoint (sub-layer 1-6, separate from the `0755` toolchain dirs). The `Makefile` wires a fourth named volume `dotfiles_gnupg` exactly like `dotfiles_cargo` / `dotfiles_rustup` / `dotfiles_mise`. `.chezmoiignore` excludes `.local/share/gnupg`. gpg-agent / pinentry run runtime, on-demand only (no agent baked into the entrypoint); gpg runs on defaults (no chezmoi-managed `gpg.conf` / `gpg-agent.conf` in this phase).

**Tech Stack:** Podman ≥ 4.0 / BuildKit, Manjaro base image, Arch `core` `gnupg` + `pinentry`, chezmoi, Python 3.11+ tomllib (existing `make gen-deps`).

## Global Constraints

- **Secret-free image** (spec 13 / spec 20 I4): no GPG key material is baked into any image layer. The Layer 1-6 directory is empty at build time; the keyring lives only in the runtime `dotfiles_gnupg` named volume. The build-prepass (Stage 2) never touches `gnupg`.
- **All packages from `packages.toml`** (spec 20 I5/I8): `gnupg` and `pinentry` are declared in `dependencies/packages.toml`; `dependencies/layer_1/pacman.txt` is generated by `make gen-deps`, never hand-edited.
- **No secret-store daemon** (design S9): `gnome-keyring` / a `libsecret` *service* is not installed or started. `libsecret` arrives only as a hard dependency of the `pinentry` package (library, not the service), preserving the Bitwarden-only secret model (spec 13 §2 Tier 1).
- **Non-root, `--userns=keep-id`** (spec 20 I2/I7): the Layer 1-6 directory is created owner-correct (`HOST_UID:HOST_GID`) so the runtime `${USERNAME}` can write the keyring and gpg's strict `0700` requirement is satisfied.
- **`USERNAME` defined in `.env`** at repo root (existing). `make build` / `make up` abort when unset.
- **`GNUPGHOME` is already set** by `dot_zshenv.tmpl` (`export GNUPGHOME="${GNUPGHOME:-$XDG_DATA_HOME/gnupg}"`); this plan does **not** edit `dot_zshenv.tmpl`.

## Task 2: Bake the `GNUPGHOME` mountpoint in Containerfile Layer 1-6

**Files:**
- Modify: `container/Containerfile` (Layer 1-5 area)

**Interfaces:**
- Consumes: `HOST_UID` / `HOST_GID` / `USERNAME` build-args (already `ARG`'d in Layer 1-1); the `USER root` … `USER ${USERNAME}` bracket of Layer 1-5.
- Produces: `/home/${USERNAME}/.local/share/gnupg` exists, `0700`, owner `HOST_UID:HOST_GID` before any runtime mount — so Podman does not root-create an absent mountpoint at `make up` (the `/home` re-own failure mode documented in Layer 5-2), and gpg's strict `0700` requirement on `GNUPGHOME` is satisfied.

- [ ] **Step 2.1: Insert the Layer 1-6 `install -d` for gnupg**

In `container/Containerfile`, find the Layer 1-5 block:

```dockerfile
# Layer 1-5: XDG-compliant home subdirectories
#
# Owner-correct directories must exist so the four runtime mounts (one bind
# for chezmoi source, three named volumes for cargo/rustup/mise) attach to
# a properly-owned ~/.local/share/ parent. install -d as root, then return
# to ${USERNAME} so subsequent stages stay non-root.
USER root
RUN install -d -o ${HOST_UID} -g ${HOST_GID} -m 0755 \
    /home/${USERNAME}/.local \
    /home/${USERNAME}/.local/share \
    /home/${USERNAME}/.local/share/cargo \
    /home/${USERNAME}/.local/share/rustup \
    /home/${USERNAME}/.local/share/mise \
    /home/${USERNAME}/.local/share/chezmoi
USER ${USERNAME}
```

Replace it with (add the Layer 1-6 `RUN` between the existing `RUN` and the closing `USER ${USERNAME}`, and extend the block comment to mention the gnupg mountpoint):

```dockerfile
# Layer 1-5: XDG-compliant home subdirectories
#
# Owner-correct directories must exist so the five runtime mounts (one bind
# for chezmoi source, four named volumes for cargo/rustup/mise/gnupg)
# attach to a properly-owned ~/.local/share/ parent. install -d as root,
# then return to ${USERNAME} so subsequent stages stay non-root.
USER root
RUN install -d -o ${HOST_UID} -g ${HOST_GID} -m 0755 \
    /home/${USERNAME}/.local \
    /home/${USERNAME}/.local/share \
    /home/${USERNAME}/.local/share/cargo \
    /home/${USERNAME}/.local/share/rustup \
    /home/${USERNAME}/.local/share/mise \
    /home/${USERNAME}/.local/share/chezmoi

# Layer 1-6: GNUPGHOME mountpoint (named volume dotfiles_gnupg).
#
# 0700 is mandatory for GNUPGHOME (gpg enforces strict permissions on its
# home and refuses otherwise). Kept as a separate RUN from the 0755
# toolchain dirs above so the mode is distinct and the intent is obvious.
# Owner-correct so Podman does not root-create an absent mountpoint at
# `make up` (the /home re-own failure mode, see Layer 5-2). The directory
# is empty at build time; no key material is baked into the image
# (spec 20 I4 / spec 13 I-S4 — the keyring lives only in the runtime
# named volume).
RUN install -d -o ${HOST_UID} -g ${HOST_GID} -m 0700 \
    /home/${USERNAME}/.local/share/gnupg
USER ${USERNAME}
```

- [ ] **Step 2.2: Build the image and confirm Layer 1-6 runs**

Run:
```bash

```
Expected: build succeeds; the Layer 1-6 `install -d` runs without error (the build does not mount the named volume, so the directory is created empty in the image layer).

- [ ] **Step 2.3: Verify the baked directory is `0700` and empty in the image**

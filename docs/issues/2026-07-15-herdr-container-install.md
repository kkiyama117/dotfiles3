# Install `herdr` during container build

**Date:** 2026-07-15
**Status:** closed (see [result-log](2026-07-15-phase-herdr-container-install.md))
**Related:** [design](../specifications/implementations/2026-07-15-herdr-container-install-design.md), [plan](../plans/2026-07-15-herdr-container-install-impl.md), spec 02, spec 20, spec 21

## Context

`herdr` is a terminal workspace manager for AI coding agents ([https://herdr.dev](https://herdr.dev)). It is currently listed as `migrated` in `dependencies/packages.toml`:

```toml
name = "herdr"
manager = "migrated"
layer = -1
has_configs = true
description = "multiplexer with AI agents"
```

This means its chezmoi-managed config directory (`~/.config/herdr/`) is retained in the dotfiles source, but the `herdr` binary is **not installed in the container**. The host installs it as a prebuilt static-pie ELF binary from GitHub releases (`ogulcancelik/herdr`), currently at version 0.7.3, with the stable manifest served from `https://herdr.dev/latest.json` and the preview manifest from `https://herdr.dev/preview.json`. The preview manifest includes SHA256 checksums per platform asset.

The container already installs other single-file, curl-bootstrapped infrastructure in the `toolchain` stage (Layer 3), most notably `cargo-binstall` (Layer 3-6), with pinned version + SHA256 verification (spec 20 I-INFRA1 / I-CARGO1). `herdr` fits the same pattern: a prebuilt binary with no build-time dependencies other than a download + integrity check.

## Problem

Provide `herdr` in the container image so that AI-agent workspace management works inside the container without requiring manual installation after `make up`. The install must preserve the existing secret-free image posture and the rule that packages must be declared in `packages.toml` and generated by `make gen-deps`.

## Acceptance criteria

1. `herdr` entry in `dependencies/packages.toml` changes from `manager = "migrated"`, `layer = -1` to `manager = "custom"`, `layer = 3`; `make gen-deps` regenerates `docs/specifications/02-installed-programs.md` AUTO-GEN block so `herdr` moves from Layer -1 to the Layer 3 `custom` section.
2. Containerfile installs `herdr` as a prebuilt binary in the `toolchain` stage with a pinned version and SHA256 verification, placing the executable at `~/.local/bin/herdr` (which is on `PATH` via `.zshenv`).
3. After `make up`, `podman exec <container> zsh -ic 'herdr --version'` exits 0 and prints a version string.
4. `~/.config/herdr/` remains chezmoi-managed; the runtime `chezmoi apply` (entrypoint) continues to render the config directory without regression.
5. No secret material is baked into any image layer. `herdr`'s config templates contain only public preferences (theme, keybindings, UI flags); the Bitwarden-only secret model (spec 13 §2 Tier 1 / spec 20 I4) is preserved.
6. `docs/specifications/02-installed-programs.md`, `docs/specifications/20-container-rules.md`, and `docs/specifications/21-container-build-flow.md` are updated to record the new Layer 3 `custom` entry and the curl-bootstrap install path.

## Notes

- `herdr` is a 17 MB static-pie binary, so the install is a single-file download; no compiler, mise-managed runtime, or package manager is required.
- The install method mirrors `cargo-binstall`: download a pinned release asset, verify SHA256, `chmod +x`, and move into `~/.local/bin/`. The `preview.json` manifest provides the SHA256 for the `linux-x86_64` asset; if the stable manifest does not include a checksum, the SHA256 can be pinned manually like `cargo-binstall`.
- `herdr` supports both a `stable` and `preview` update channel (set in `~/.config/herdr/config.toml`). The host currently uses `preview`, but the build-time install should pin one channel/version for reproducibility. The default issue design uses the preview build because the preview manifest ships SHA256 hashes; switching to stable later is a one-line version/SHA bump.
- Runtime updates via `herdr update` are not executed during build; the binary stays at the pinned version until the next image rebuild.
